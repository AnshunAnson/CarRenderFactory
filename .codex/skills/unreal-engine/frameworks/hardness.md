# Hardness Framework (执行强度分级)

> **权威源**: 本文件定义 Hardness 分级规则，SKILL.md 为入口索引
> **版本**: 1.0

---

## 概述

Hardness Framework 定义了任务执行的强度分级，控制：
- 委托深度
- 并行度
- 验证严格度
- 验收门禁

---

## Hardness 级别定义

### L (Low) - 低复杂度 / 低风险

**特征**:
- 单文件或窄范围修改
- 无多人/复制/GAS 关键路径
- 低不确定性

**执行规则**:
- 默认: 无委托
- 主 Agent 直接处理请求
- 如果 API/插件行为不确定，临时委托 `document-specialist` 查询官方文档

**验证要求**:
- 快速验证
- 假设声明即可，无高风险路径

---

### M (Medium) - 中等复杂度 / 标准风险

**特征**:
- 多文件功能/修复
- 无 H 级触发条件
- 调试、评审、验证、研究密集型工作

**执行规则**:
- 必须运行两条车道 (可并行):
  - `explore`: 代码/资产/配置发现
  - `executor`: 实现或变更计划起草
- 必须在执行后运行 `verifier`
- 禁止同车道自我批准

**验证要求**:
- PIE + 独立检查
- 残余风险列表

---

### H (High) - 高复杂度 / 高风险

**特征**:
- 跨模块多文件重构
- 复制/GAS/网络权威关键路径变更
- 插件或版本化 API 高度不确定
- 安全/可靠性敏感行为

**执行规则**:
- 必须流水线: `planner -> architect -> executor -> verifier -> code-reviewer`
- 必须拆分为至少两条独立并行子任务 (如: 复制车道 + GAS 车道)
- 如果 SDK/插件/API 不确定，强制 `document-specialist`

**验证要求**:
- 客户端/服务器检查
- 复制/GAS 断言
- 未解决列表 (如有)

**失败策略 (team-fix policy)**:
```
max_attempts = 3
max_duration = 45m

提前停止条件:
- 关键检查仍然失败
- 重复相同失败签名
- 验证证据缺失两次

达到限制/停止时:
- 设置 Status: BLOCKED
- 返回未解决风险 + 下一步负责人/行动
```

---

## Hardness 选择规则

### 选择 H 当满足任一条件:
- 跨模块多文件重构
- 复制/GAS/网络权威关键路径变更
- 插件或版本化 API 高度不确定
- 安全/可靠性敏感行为

### 选择 M 当满足任一条件:
- 多文件功能/修复 (无 H 触发条件)
- 调试、评审、验证、研究密集型 UE 工作

### 选择 L 当满足所有条件:
- 低风险 Q&A、单命令辅助、窄范围本地建议
- 无广泛重构且无关键多人/复制路径

---

## AgentHardness 合规规则 (严格)

1. **Hardness 级别控制**:
   - 委托深度
   - 并行度
   - 验证严格度
   - 验收门禁

2. **禁止事项**:
   - 禁止用 L 级验证执行 H 级工作
   - 禁止无明确证据理由降级 Hardness 以节省时间

3. **动态升级**:
   - 如果任务中观察到风险增加，立即升级 `L->M` 或 `M->H`
   - 重新规划团队

4. **无法界定风险时**:
   - 返回 `Status: BLOCKED`

## 级别与输出要求对应

| Hardness | 必需输出字段 | 验证证据要求 |
|----------|-------------|-------------|
| **L** | Hardness, Status | 假设声明，无高风险路径 |
| **M** | 完整输出契约 + Review Agent Findings + Context Experience | PIE + 独立检查 + 残余风险列表 |
| **H** | 完整输出契约 + Review Agent Findings + Context Experience + Skill Optimization Proposals | 客户端/服务器检查 + 复制/GAS 断言 + 未解决列表 |
